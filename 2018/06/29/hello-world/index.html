node压测

node压测

压测类型

  • 接口压测
  • 页面压测（全链路压测）

名词解释

  • tps：每秒请求数，应用性能的重要度量指标之一
  • concurrent: 并发数
  • RT：响应时间

压测前准备

  • 在k2中搭建一个新的namespace
  • 配置好对应的后端压测环境
  • 选择全新的域名专为压测使用（否则会影响到测试环境）

小点补充

  • 一般情况下压力测试客户端接收到的数据量一定会远大于发送出去的数据，但不一定
  • 有些特例，比如有些post接口，参数可能是个列表，但是返回值可能是bool类型。这样就数据量传出去很大，返回很小
  • 由于线上测试环境的数据量有较大差别（线上数据量较全），以及线上测试环境的机器性能有较大差别，一般以线上高峰数据进行倒推，测试环境必须达到线上要求单台的tps峰值

测试工具

  • ab（apache自带的一款功能强大的测试工具）
  • jmeter
  • smurf

ab 测试方法

一般就用来自己性能优化的时候测试使用

  1. 主要参数
// -c 
concurrency "并发量，一次产生的请求个数，默认是一次一个"
// -t
timelimit "测试所进行的最大秒数"
// -n
"总共请求个数"
// -C
cookie-name=value "对请求附加一个Cookie"
  1. 测试
ab -n 30000 -c 1000 http://test
// 设置最大请求数和并发请求数

ab -t 60 -c 100 http://test
// 设置请求时间和并发请求数
  1. 测试结果主要参数解读
Concurrency Level: 1000
// 测试的并发数
Time taken for tests: 11.846 seconds
// 整个测试持续的时间
Complete requests: 4000
// 完成的请求数量
Failed requests: 0
// 失败的请求数量
Write errors: 0
Total transferred: 204586997 bytes
// 整个过程中的网络传输量
HTML transferred: 203479961 bytes
// 整个过程中的HTML内容传输量
Requests per second: 337.67 [#/sec] (mean)
// 最重要的指标之一，相当于LR中的每秒事务数，后面括号中的mean表示这是一个平均值
Time per request: 2961.449 [ms] (mean)
// 最重要的指标之二，相当于LR中的平均事务响应时间，后面括号中的mean表示这是一个平均值
Time per request: 2.961 [ms] (mean, across all concurrent requests)
// 每个连接请求实际运行时间的平均值
Transfer rate: 16866.07 [Kbytes/sec] received
// 平均每秒网络上的流量，可以帮助排除是否存在网络流量过大导致响应时间延长的问题
  1. 注意点
  • ab命令在一般系统上面做测试时候，一般并发不能超过1024个，其实是因为因为系统限制每个进程打开的最大文件数为1024
  • 最大请求数量不要超过五万个

jmeter测试方法

一般用来测试单接口 会产出结果图表

  1. 首先你需要下载一个jmeter

  2. 下载安装一个插件 将它移到jmeter\lib\ext下 然后重启jmeter
    jmeter-plugins-graphs-basic-2.0.jar

  3. 打开下载的jmeter 在bin目录下找到jmeter.sh文件 用终端打开可启动jmeter

  4. 右键点击测试计划(Test Plan)添加->Threads->Threads Group(线程组) 线程组是用来定义虚拟用户数量、用户加载方式和时间调度的组件。 所有请求都是运行于线程组之下的，这个组件是必须的。
  1. 添加->sample->HTTP请求

你也可以采用更接近实际情况的分段并发模拟 右键测试计划->添加->Threads->jp@gc - Stepping Thread Group

  1. 配置响应断言 添加->断言->响应式断言

  1. 添加->监听器->聚合报告 （和以下两个为压测之后看结果用）

  2. 添加->监听器->jp@gc-Transactions per second

  3. 添加->监听器->jp@gc-Response Times Over Time

  4. 点击绿色小三角 开始

  5. 查看结果
