node日志

日志打印规范

目的

  1. 本地调试代码
  2. 线上问题跟踪
  3. 场景还原
  4. 监控调优

日志级别

  • DEBUG（调试）：本地开发调试日志。

    一般来说，在系统实际运行过程中，不会输出该级别的日志。因此，开发人员可以打印任何自己觉得有利于了解系统运行状态的东东。不过很多场景下，过多的 DEBUG 日志，并不是好事，建议是按照业务逻辑的走向打印。打个比方，打印日志就像读书时划重点，如果导出都是重点，也就失去了重点。

  • INFO（通知）：记录系统运行状态等关联信息。

    该日志级别，常用于反馈系统当前状态给最终用户。所以，在这里输出的信息，应该对最终用户具有实际意义，也就是最终用户要能够看得明白是什么意思才行。

  • WARN（警告）：表示系统模块发生问题，但并不影响系统运行。

    此时，进行一些修复性的工作，还能把系统恢复到正常的状态。

  • ERROR（错误）：系统抛错。

    主体系统核心模块正常工作，需要修复才能正常工作。 就是说可以进行一些修复性的工作，但无法确定系统会正常的工作下去，系统在以后的某个阶段，很可能会因为当前的这个问题，导致一个无法修复的错误（例如宕机），但也可能一直工作到停止也不出现严重问题。

日志内容

  • 时间，包含时区信息和毫秒（默认记录）
  • 日志级别，例如 debug、info 以及 warn、error 根据不同方法区分。
  • 会话标识，用以记录触发的客户端或用户，比如登陆账号，seesion 信息等。
  • 功能标识，方便日志搜索，并跟踪指定功能的完整轨迹。
  • 精炼的内容，包括场景信息（谁，什么功能等），状态信息 (开始，中断，结束) 以及重要参数。
  • 其他信息，其它可能的有用信息包括：版本号，线程号等等。

打印时机

日志级别 时机 对应 api
DEBUG 调试日志。调试使用。 log.debug
INFO 业务日志。记录业务的主流程的走向。 log.info
WARN 警告日志。一般来说，发生对整个系统没什么影响的异常时，可以打印该级别的日志。 log.warn
ERROR 错误日志。级别比较高，如果发生一些异常，并且任何时候都需要打印时使用。 log.error

注意事项

  1. 全打印等于没打印
  2. 打印日志也会有性能损耗，请慎重
  3. 打印日志后，务必配置 grail 报警，用于接收报警信息
  4. 千万不要将 console.log 等代码发布到生产环境，由于 console.log 日志由 pm2 收集在 ~/.pm2 目录下，而 ~/ 目录又有大小限制，使得服务器会报警，请务必完全使用日志文件代替
  5. 仅收集传入『错误对象』的有限字段（比如：message 和 stack），自定义字段请自行转为字符串，并加入 message 字段中
